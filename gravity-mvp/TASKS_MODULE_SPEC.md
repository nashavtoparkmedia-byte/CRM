# Спецификация и Архитектурный Дизайн: Модуль «Задачи» (Tasks)

## 1. Анализ задачи (Understanding Summary)
* **Что мы строим:** Операционный раздел «Задачи» для CRM с тремя синхронизированными представлениями (Список, Время, Доска). Это не три разных модуля, а три режима просмотра и работы с одной и той же сущностью Task. Изменение задачи в одном месте мгновенно отражается во всех.
* **Зачем это нужно:** Создать единую операционную среду для проактивной работы с водителями. Менеджер должен с одного взгляда понимать: что происходит с водителями, кому нужно написать, где просрочка и есть ли ответ.
* **Для кого это:** Для менеджеров парков и руководителей. Интерфейс и продуктовая логика проектируются под реальный рабочий контур до 100 активных задач в моменте, без избыточного усложнения под гипотетические большие очереди. Backend может иметь запас по масштабированию.
* **Ключевые требования/ограничения:**
  * **Глубокая интеграция:** Разделы Задачи и Чаты работают как единая операционная среда. Чат отвечает на вопрос "что написал водитель", задача — "что с этим сделать".
  * **Триггеры (Смешанный подход):** Push (реактивно) для моментальных событий, Cron (раз в час) для временных условий, плюс ночная `reconciliation` сверка.
  * **Database-level Anti-duplication:** Для auto-задач дубли запрещаются на уровне БД. Для manual/chat задач применяется soft-dedupe на уровне UI/сервиса без жесткого запрета на создание.
* **Явные "не-цели" (Non-goals):** Это не шаблонная тяжеловесная kanban-доска, не изолированный от чатов модуль и не таблица ради таблицы.

## 2. Предположения (Assumptions)
* **Архитектура UI:** Упор на «живость» (real-time обновления), быстрый переход к действию и плотный контекст, а не на сложную абстрактную энтерпрайз-виртуализацию.
* **Автоматика:** Логика первичной auto-задачи и follow-up определяется политикой триггера: follow-up либо усиливает существующую активную задачу, либо создает новую сущность с новым dedupe_key.
* **Аудит:** Вся история auto-задач надежно логируется (причина, время срабатывания, payload условия, была ли задача reopened, было ли отправлено автоматическое сообщение, был ли получен ответ).

## 3. Финальный Архитектурный Дизайн

### 3.1 Архитектура и Потоки Данных (Data Flow)
**Серверная часть (Backend)**
* Реализуется на PostgreSQL + Prisma с классической state-based моделью.
* Таблица `Task` хранит текущее состояние, таблица `TaskEvent` — append-only историю системных событий и переходов.
* **Единый контракт:** API реализуется единообразно по контракту. Все представления (`list`, `timeline`, `board`) используют **одну и ту же базовую DTO-модель** `Task`. Список, Время и Доска являются только разными представлениями одной и той же сущности Task и не вводят собственные независимые модели данных. Идентичность Task и набор обязательных базовых полей не зависят от представления UI.
* **Разделение DTO:** Базовая задача компактна. Тяжелые данные (`TaskHistoryDTO`, `DriverContextDTO`, `ChatPreviewDTO`, `TriggerDetails`) не раздувают списки и загружаются on-demand в карточке задачи.

**Слой сети и кэширования (Query Layer)**
* React Query используется как инфраструктура для: initial fetch, reconnect sync, background refetch, stale revalidation, retry / network resilience, server reconciliation. 
* `useInfiniteQuery` не ставится по умолчанию для активного контура; базовая загрузка — обычный лимитированный `useQuery`. Инфинит используется только для истории/закрытых задач.

**Слой живого UI (Store Layer)**
* Поверх Query Layer работает нормализованный Store (Zustand). Единый нормализованный Store хранит как минимум: `tasksById`, `taskIds`, `currentView`, `filters`, `sort`, `selectedTaskId`, `selectedDriverId`, `timelineRange`, и UI state текущего рабочего контура (например, открытые панели, режим выбора, локальное состояние текущего вида).
* Корневой слой гидрирует Store, а не пересоздает его при рендерах.
* Все виды (Доска, Список, Время) — глупые компоненты, работающие через derived selectors. Они не делают свои API-вызовы, не выполняют локальную бизнес-фильтрацию и не хранят дублирующий стейт задач.

**Real-time и Мутации (Optimistic Updates)**
* **Действия пользователя:** Завершение, смена статуса или drag-and-drop сразу оптимистично обновляют Store. Уходит API-вызов. При ошибке API выполняется rollback или server reconciliation. Если сервер возвращает валидное итоговое состояние, отличающееся от optimistic state, клиент применяет server truth.
* **WebSocket:** Адресно передает события типа `upsert`, `patch`, `reopen`, `archive`, `remove-from-active-view`, `update-linkage`, `append-event` и обновляет store без грубой инвалидации всего кэша.

### 3.2 Схема БД и Механика Защиты от дублей
**Dedupe Model в `Task`**
Для уверенной работы авто-задач фиксируем разделение логики причин и ключей консолидации:
* `trigger_type` — бизнес-семантика причины (например, `driver_inactive`).
* `trigger_key` — конкретное вычисление или дата.
* `dedupe_key` — технический ключ консолидации и дедупликации активной задачи.
* **Database Level Protection:** На уровне PostgreSQL ставится частичный уникальный индекс, применяемый только к активным auto-задачам:
  ```sql
  CREATE UNIQUE INDEX unique_active_auto_task 
  ON tasks (driver_id, source, dedupe_key) 
  WHERE is_active = true AND source = 'auto';
  ```
  Поле source обязательно различает как минимум auto, manual, chat, поскольку политика дедупликации и жизненный цикл задачи зависят от источника создания.

**Follow-up Policy & Conflict Resolution**
При конфликте вставки применяется policy-based conflict resolution:
* `no-op`
* `patch` существующей задачи
* `reopen` существующей задачи
* усиление существующей задачи (например, повышение priority, перенос due_at, обновление metadata)
* либо создание новой сущности с новым `dedupe_key`, если так определено trigger policy.

Конкретный сценарий conflict resolution определяется trigger policy и текущим состоянием активной задачи. Follow-up policy не определяется “на лету” воркером, а задается явно в настройке конкретного trigger policy.

### 3.3 Интеграция с Чатами (Chat Sync)
Чаты и Задачи формируют **единый рабочий контекст**, а не живут как два разрозненных модуля.

**Контекстный Блок:** 
В правой панели Чатов отображается компактный блок задач водителя: статистика, до 3-5 активных задач, кнопка "Создать задачу", и кнопка "Открыть все задачи". Блок читает данные из того же Store/API, что и основной модуль Задач.

**Сохранение Origin (Создание из чата):**
Задача, созданная из чата, обязательно получает `source = 'chat'`, `chat_id` и, при наличии, `origin_message_id`. Желательно также сохранять `origin_excerpt` для удобного отображения контекста в карточке задачи. При наличии также желательно сохранять `origin_created_at`. Карточка задачи обязательно содержит кнопки «Перейти к сообщению-источнику» и «Открыть чат».

**Soft-dedupe для manual/chat задач:**
Система мягко предупреждает о наличии похожей активной задачи по этому водителю, с учетом как минимум типа задачи и близости срока, предлагая открыть её или осознанно создать новую, не запрещая создание жестко.

**Policy-based Переходы & Флаг новой реакции:**
Переходы статусов по событиям чата происходят по бизнес-правилам, а не безусловно. 
* Исходящее релевантное сообщение обновляет `last_outbound_message_at` и при необходимости переводит задачу в `waiting_reply`. 
* Входящее релевантное сообщение обновляет `last_inbound_message_at`, выставляет `has_new_reply = true` и только при наличии policy-rule может менять статус задачи.
* Метки `last_outbound_message_at` и `last_inbound_message_at` используются как часть рабочего контекста задачи и логики follow-up.
* Любые значимые chat-related события, влияющие на состояние задачи, фиксируются в `TaskEvent`.

## 4. Журнал Принятых Решений (Decision Log)

1. **Single Task Identity Across Views:** Список, Время и Доска работают поверх одной и той же сущности Task без отдельных client/server моделей под каждый режим.
2. **Архитектура UI State:** Гибрид Unified Store как живая клиентская модель + React Query как сетевой слой.
3. **DTO Separation:** Единый базовый `Task DTO` используется для всех представлений; сервер не вводит отдельные несовместимые DTO-модели задач для отдельных представлений.
4. **Отказ от бесконечного скролла в активе:** Активный слой работает через лимитированный fetch, а не `useInfiniteQuery`.
5. **Отказ от event-sourcing / CQRS как базовой архитектуры:** Полноценный event-sourcing / CQRS как основа модуля не используется. Применяется классическая state-based модель Task + append-only аудит в TaskEvent. Причина: YAGNI.
6. **Dedupe Key + Partial DB Index:** Внедрение технического `dedupe_key` и частичного уникального индекса на уровне БД (`is_active = true AND source = 'auto'`).
7. **Chat Flag `has_new_reply`:** Замена безусловного перехода "waiting_reply -> todo" на выставление индикатора `has_new_reply` и Policy Rules. 
8. **Chat Origin Linkage:** Задачи, созданные из чатов, сохраняют связь с `chat_id` и, при наличии, `origin_message_id`, чтобы поддерживать обратный переход в коммуникационный контекст.
