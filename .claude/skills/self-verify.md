# Self-Verify — Самопроверка перед отчётом

## Правило
НИКОГДА не говорить пользователю "готово, проверь". Вместо этого — самому проверить, написать тесты, убедиться что работает, и отчитаться с доказательствами.

## Алгоритм после каждой задачи

### 1. Build проверка
```bash
cd gravity-mvp && npm run build
```
Если ошибки — починить, не отчитываться.

### 2. Написать тест для изменений

В зависимости от типа изменений:

**API endpoint / Backend логика:**
- Создать `scripts/test_<feature>.js` в gravity-mvp
- Тест должен вызвать endpoint / функцию с реальными данными
- Проверить response status, body, побочные эффекты в БД
- Проверить edge cases (невалидный input, пустые данные, дубликаты)

```js
// Пример структуры теста
async function test() {
  console.log('Testing <feature>...');

  // Setup
  const testData = { ... };

  // Act
  const response = await fetch('http://localhost:3002/api/...', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(testData)
  });

  // Assert
  const data = await response.json();
  console.assert(response.status === 200, `Expected 200, got ${response.status}`);
  console.assert(data.success === true, `Expected success, got ${JSON.stringify(data)}`);

  // Verify side effects
  // ... проверить БД, файлы, etc.

  // Cleanup
  // ... удалить тестовые данные

  console.log('✅ All tests passed');
}

test().catch(e => { console.error('❌ Test failed:', e.message); process.exit(1); });
```

**Pipeline / AI изменения:**
- Отправить тестовое сообщение через webhook
- Проверить MessageEventLog → AiDecisionLog цепочку
- Убедиться что intent/decision корректны

**UI компонент:**
- Запустить dev server если не запущен
- Открыть страницу через preview или браузер
- Сделать скриншот
- Проверить что элемент отображается, нет ошибок в консоли

**Prisma / БД изменения:**
- Проверить что миграция применилась: `npx prisma migrate status`
- Тест CRUD операций для новой/изменённой модели
- Проверить что существующие данные не сломались

**Scraper изменения (MAX / Yandex):**
- Проверить что сервер запускается без ошибок
- Тест парсинга на фиксированных данных
- Проверить webhook отправку

### 3. Запустить тест
```bash
node scripts/test_<feature>.js
```
- Если тест упал — починить код, перезапустить тест
- Повторять пока не пройдёт

### 4. Проверить что не сломал существующее
- Если менял API — проверить что другие endpoints отвечают
- Если менял БД — проверить что CRM страницы загружаются
- Если менял pipeline — проверить что старые сообщения всё ещё обрабатываются

### 5. Очистка
- Удалить тестовые данные из БД
- Оставить тестовый скрипт (может пригодиться)

### 6. Отчёт пользователю

Формат отчёта:
```
Что сделано: <краткое описание>
Изменённые файлы: <список>
Тесты: <что проверено>
Результат: ✅ Build OK, тесты пройдены
```

Если есть скриншот UI — приложить.
Если тест показал edge case — упомянуть.

## Чего НЕ делать
- НЕ говорить "готово, проверь сам"
- НЕ отчитываться без запуска build
- НЕ пропускать тесты для "мелких" изменений — даже однострочный фикс тестировать
- НЕ удалять тестовый скрипт (пользователь может захотеть перезапустить)
- НЕ писать тесты которые всегда проходят (assert реальные значения, не просто "no error")
