# MAX Debug — Отладка MAX скрапера

## Что делать

### 1. Проверить доступность скрапера
```js
fetch('http://localhost:3005').then(r => r.json())
```
Также проверить `/status` если есть.

### 2. Проверить статус сессии
Прочитать файлы состояния в `max-web-scraper/`:
- `session/` — данные сессии Playwright
- `user_data/` — профиль браузера

Проверить endpoint состояния (если есть `/api/session/status` или аналог).

### 3. Проверить known_chats.json
Прочитать `max-web-scraper/known_chats.json`:
- Сколько чатов
- Когда последнее обновление
- Есть ли дубликаты

### 4. Проверить dedup кэш
Прочитать `max-web-scraper/last_seen_dedupe.json`:
- Размер кэша (количество записей)
- Самая старая и новая запись
- Предупредить если кэш > 10000 записей (утечка памяти)

### 5. Проверить логи WS
Если скрапер запущен — посмотреть последние строки логов:
- Есть ли ошибки reconnect
- Последний opcode 19 (auth) — когда был
- Есть ли "CRM webhook вернул 500"

### 6. Проверить Anti-Ghost логику
Прочитать `.claude/knowledge/max_chat_merging.md` для контекста.
Проверить что webhook fallback работает:
- В known_chats есть записи с телефонами
- Нет чатов с пустым senderId

### 7. Отчёт

```
MAX Web Scraper Debug
═══════════════════════
Сервер:          ✅ Работает на :3005 / ❌ Не отвечает
Сессия:          ✅ Активна / ⚠️ Нужен QR / ❌ Нет сессии
WS соединение:   ✅ Подключен / ❌ Отключен
Known chats:     N чатов (последнее обновление: дата)
Dedup кэш:       N записей (⚠️ если > 10000)
Последнее сообщение: timestamp
Ошибки:          список если есть
```

Если есть проблемы — предложить конкретные шаги для исправления.
